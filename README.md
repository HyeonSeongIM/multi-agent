# Multi-agent

## 도입 배경
디자인, PRD, 프론트, 데이터 분석, 유지보수 등등 다양한 업무들을
하기위해 많은 사람들이 붙어야합니다. 이런 점들을 에이전트가 일부 작업을 해준다면
본인이 직접 상상한 앱, 웹 아이디어등 다양한 요소에서 개발을 할 수 있다고 생각하여 임하게 되었습니다.

## 왜 OpenClaw 인가
2026.05.22 기준으로 가장 최신 AI-Agent 툴은 헤르메스 에이전트가 가장 유명합니다.

하지만 저는 OpenClaw를 사용할려고 합니다. 이유는 다음과 같습니다.

[1] 기억 저장 필요 여부<br>
기억을 저장할 필요가 없다고 생각합니다. 현재 제가 구축하는 에이전트들은
agents.md 파일을 주고 작동하게 하면 된다고 생각합니다. 추후 기억 저장이 필요하다고 생각이 들면
헤르메스 에이전트로 마이그레이션 해보겠습니다. 

[2] 정해진 용량<br>
현재 맥미니 또는 용량이 큰 미니 저장소가 아닌 라즈베리파이로 운영할려고 합니다.
따라서 헤르메스 에이전트는 기억을 해야하기 때문에 저장소용량이 크게 필요하다고 생각하여
현재는 오픈클로로 시작했습니다.

## 각 에이전트 별 역할
### 관리 측면
로그 분석 Agent<br>
S3에서 앱 1~4의 로그 파일을 주기적으로 가져와서 에러율, 응답시간 이상, 반복 에러 패턴을 감지해요. 결과는 레포트 Agent로 전달하고, 임계값 초과시 긴급 알림 Agent도 동시에 트리거해요.

<br>데이터 분석 Agent<br>
DAU, 매출, 전환율 같은 비즈니스 지표를 S3에서 읽어서 트렌드를 분석해요. 로그 분석이 "기술적 이상"을 보는 거라면 얘는 "비즈니스 이상"을 봐요. 결과는 레포트 Agent로만 전달해요.

<br>모니터링 Agent<br
서버 CPU, 메모리, 디스크, 가용성을 상시 체크해요. 헬스체크 엔드포인트를 주기적으로 찌르는 역할도 해요. 배포 완료 후 자동으로 헬스체크 실행하는 것도 이 Agent가 담당해요. 이상 감지시 긴급 알림 Agent 트리거.

<br>긴급 알림 Agent<br>
로그 분석, 모니터링에서 임계값 초과 신호가 오면 레포트 Agent를 거치지 않고 즉시 나한테 Telegram으로 쏴요. 필터 없이 무조건 즉시 전송이 원칙이에요.

<br>레포트 Agent<br>
모든 분석 Agent의 결과를 취합하고 중요도 판단해서 필터링하는 게이트키퍼예요.

### 배포 측면
배포 알림 Agent<br>
Github Actions에서 배포 완료/실패 이벤트를 받아서 나한테 알림을 줘요. 배포 성공은 일반 등급(일일 요약), 배포 실패는 긴급 등급(즉시 전송)으로 처리해요. 배포 완료 후 모니터링 Agent한테 헬스체크 실행 신호도 보내요.

### 개발 측면
프론트엔드 Agent<br>
백엔드 코드 커밋 후 자동 생성된 명세서(OpenAPI)를 읽고 변경된 부분만 파악해서 프론트엔드 코드를 수정해요. 수정 완료 후 Github API로 브랜치 푸시 + PR 자동 생성하고 나한테 "PR 생성됨" 알림을 줘요.
